# Bandmate — Conventions & Architecture

_Last updated: 2026-07-21 (Phase 5b — push notifications: pg_net + Edge Functions, connection-gated triggers, Expo push tokens). Update this file whenever a phase introduces or changes anything below — it should never go stale._

## Stack
- Expo SDK ~57.0.6, React Native 0.86.0, React 19.2.3, TypeScript ~6.0.3, strict mode on.
- **NativeWind v4** — Tailwind utility classes for RN styling, kept for a single mental model across future web/native work.
- **React Navigation v7** (`@react-navigation/native-stack` + `@react-navigation/bottom-tabs`) — stack-inside-tabs composition; see Navigation & state below.
- **Supabase** (`@supabase/supabase-js`) — Postgres + Auth + Storage backend, single project (`ktfrsgffgzepadmoryps`, ref also in `.env`).
- **Managed workflow** — no `ios`/`android` folders checked in (gitignored, regenerated on demand). All native modules used so far (`expo-audio`, `expo-video`, `expo-image-picker`, `expo-image-manipulator`, `expo-video-thumbnails`, `expo-asset`) are bundled in Expo Go for SDK 57, so the app runs there directly.
  - **What would force a custom dev client**: adding any native module *not* included in Expo Go — a custom Expo Modules API module, a third-party RN library without Expo Go support, background tasks/notifications requiring native entitlements, or anything needing a custom `Info.plist`/`AndroidManifest.xml` entry beyond what `app.json` config plugins cover.
  - **This predicted exactly what Phase 5b hit**: `expo-notifications` remote push is unavailable in Expo Go on Android from SDK 53 onward (confirmed against the versioned SDK 57 docs, not assumed) — a real device + a development/EAS build is required to actually receive a push on Android. iOS's story is less restrictive but still ultimately needs a real device for genuine delivery. See Known tech debt — this app has no EAS project yet, so this is currently unverifiable either way, not just "unverified on this one platform."

## Data model
- Migrations live in `supabase/migrations/`, named `NNNN_description.sql` (`0001_init.sql`, `0002_media_feed.sql`, `0003_dev_confirm_user_email.sql`, `0004_restrict_dev_confirm_user_email.sql`, `0005_connections_delete_and_pair_constraint.sql`, `0006_storage_buckets_select_policy.sql`, `0007_media_storage_setup.sql`, `0008_availability_status_and_self_connect_check.sql`, `0009_cities.sql`, `0010_discover_profiles.sql`, `0011_messages_connection_gate_and_read_receipts.sql`, `0012_push_notifications.sql`). Applied via the Supabase MCP `apply_migration` tool (no local Supabase CLI installed — see Known tech debt) to **both** the production and test projects (see Testing) — there are two live databases now, and this is the only source of truth kept in sync between them.
  - **`0012` is the one migration file that is genuinely byte-identical across both projects** (the reverse of `0009`'s per-project real data) — the one truly per-project literal (each project's own Edge Functions base URL) is seeded into Vault via a separate, uncommitted direct-SQL statement per project instead of being embedded in the tracked file, same "real per-project value, not the tracked migration" precedent `cities`' row data set. See `push_tokens` below for the reasoning.
  - **`0010`'s live application history has extra entries beyond the one committed file**: a type bug (`<@>` returns `double precision`, not `numeric`) and a stability bug (`order by created_at desc` alone isn't a strict total order — see below) were both found and fixed via `apply_migration` calls *before* the file was ever committed, each re-running the corrected `create or replace function` under a different migration name (`discover_profiles_fix_distance_cast`, `discover_profiles_fix_search_path`, `discover_profiles_stable_order`). The committed `0010_discover_profiles.sql` reflects only the final, correct version — same "fix in place before it ships" judgment call as any normal pre-commit bug, not a new instance of the Phase 3 migration-history-drift pattern (that one was about *already-shipped* migrations silently applied outside the tracked path). Disclosed here since `list_migrations` on either project will show more entries than there are files.
- **`cities`** (Phase 4a, `0009_cities.sql`) — curated reference table for the geography foundation ahead of Phase 4b's distance/radius search: `id` (uuid pk), `city`, `state`, `lat`/`lng` (double precision, city-center approximations, not precise geocodes), `unique(city, state)`. RLS follows the same public-read-reference-data pattern as `instruments`/`genres` (`cities_read`: `for select to authenticated using (true)`) — **zero insert/update/delete policies for any client role**, so the table is writable only via migrations or direct SQL, never from the app. `profiles.matched_city_id` (nullable `uuid references cities(id) on delete set null`, added in the same migration) is the resolved link used for future distance math; `location_city`/`location_state` remain the free-text display fields and are set independently — a profile can have display text with no resolved match (`matched_city_id is null`), which is expected, not an error state.
  - **Table content (row data) is deliberately NOT part of the tracked migration** and was seeded via direct SQL instead, one-off per project — a genuine exception to "migrations are the only source of truth kept in sync." The Phase 4a city list was sourced from each project's own actual distinct `(location_city, location_state)` values at seed time (querying `pg_catalog`/`profiles` directly, not a live geocoding API), and production and test **intentionally had different rows** as a result: production had `Fort Collins, CO`; test had `Austin, TX`. Several other distinct test-project values (`Draper, IL`, `Fort Helga, WV`, `Huntersville, ID`, `New Jimcester, NV`, `Olaftown, NE`, `Rowanstead, NC`, `South Maggieville, VA`, `Stantonchester, AZ`, `Teresahaven, MD`) were skipped as obvious faker-generated fictional placenames, not real cities.
  - **Phase 4b expanded and unified the list**: `Boulder, CO` and `Denver, CO` were added to **both** projects (via an idempotent `insert ... on conflict (city, state) do nothing`, safely re-runnable), specifically because meaningful radius-search testing needs more than one real point and a genuinely close, real city pair to test a tight radius against (Boulder↔Denver is ~24 real miles apart — confirmed via the `<@>` operator itself, not assumed). Both projects now carry the same 4 cities (`Fort Collins, CO`, `Boulder, CO`, `Denver, CO`, `Austin, TX`) — a deliberate shift away from Phase 4a's "intentionally different per project" stance now that this is curated expansion rather than derived-from-live-data seeding.
  - **Still sparse**: 4 rows total is enough to exercise the picker, backfill, and radius math correctly, but not enough for rich real-world discovery variety. Growing it further (a broader static city list, or accepting rows as real users hit the "my city isn't listed" fallback over time) remains unstarted work.
  - **Backfill** (existing profiles, run once via direct SQL, not a migration): case-insensitive, trimmed exact match of `location_city`/`location_state` against `cities`; no fuzzy matching, no guessing — no match leaves `matched_city_id` null. Phase 4a result: production 1/1 matched; test 2/14 matched (the two genuine `Austin, TX` profiles), 12 unmatched. Not re-run in Phase 4b — the two new cities (Boulder/Denver) had no pre-existing profiles claiming them by free-text city/state, so there was nothing new to backfill.
- **`discover_profiles(instrument_ids, genre_ids, radius_miles, page_limit, page_offset)`** (Postgres function, `0010_discover_profiles.sql`) — backs the Discover tab. Contract, confirmed against `database.types.ts` rather than assumed:
  - **`instrument_ids`/`genre_ids` are `integer[]`, not `uuid[]`** — the original phase spec assumed uuid; `instruments.id`/`genres.id` are actually serial integers (same as `profile_instruments`/`profile_genres`'s FK columns). Caught by checking the generated types before writing the function, per the spec's own instruction not to guess. Both default to `'{}'` (empty = that category isn't filtered).
  - **Filter semantics**: within a category, OR (candidate matches if they have *any* of the given ids); across categories, AND (a non-empty `instrument_ids` and a non-empty `genre_ids` both narrow the result — confirmed by an integration test showing the combined-filter result set is a strict subset of either single-category filter's result).
  - **`security invoker`** (explicit, not just relying on the default) with `set search_path = public, extensions` — every underlying table is already authenticated-readable per this repo's RLS pattern, so no elevated privilege is needed, unlike `dev_confirm_user_email`. `EXECUTE` is revoked from `PUBLIC`/`anon`, granted only to `authenticated` — anon calls would just get RLS-filtered-empty results anyway (no elevated bypass), but this makes the restriction explicit rather than relying on that side effect.
  - **Resolves the caller via `auth.uid()` internally** — no `user_id` param exists, and the caller is always excluded from their own results.
  - **Radius**: uses `earthdistance`'s `point <@> point` operator (statute miles, no unit conversion needed), not `postgis` (unused elsewhere in this repo, would be a second redundant way to do the same math) and not `cube`'s raw `earth_distance()` (the `<@>` form is simpler and needs no `earth()`/`ll_to_earth()` wrapping). If the *caller* has no `matched_city_id`, `radius_miles` is silently ignored server-side (treated as if it weren't passed) — the Discover screen's disabled radius control is the primary guard, this is defense-in-depth. If the *caller* does have a matched city, candidates with no `matched_city_id` of their own are excluded whenever radius filtering is actually active (no way to compute their distance) but **still appear in the unfiltered case** — this asymmetry is intentional and integration-tested.
  - **`distance_miles`** in the return row is only non-null when a radius filter was actually applied — confirmed via an integration test that the same candidate gets a `null` distance on an unfiltered call and a real number on a radius-filtered one.
  - **Pagination stability**: `order by created_at desc, id asc` — the `id` tiebreaker isn't cosmetic. A batch-inserted seed run (`scripts/seed.js` inserts all profiles in one `.insert(rows)` call) gives many rows the *identical* `created_at` timestamp, and `order by created_at desc` alone is not a strict total order — without the tiebreaker, `LIMIT`/`OFFSET` pagination across separate calls was observed to return inconsistent results when ties existed (concatenating paginated pages didn't reproduce a single large-page fetch). Caught by this migration's own integration test failing, not a hypothetical — fixed at the source rather than working around it client-side.
  - **`cube`/`earthdistance` extensions enabled** (`with schema extensions`, matching where this project's other installed extensions — `pgcrypto`, `uuid-ossp` — already live) on both projects; were available-but-not-installed as of Phase 4a (`list_extensions` confirmed, not re-checked here since the spec said not to). `postgis` remains available-but-not-installed, deliberately unused (see above).
  - **No spatial index on `cities`** — the table is still small (~4 rows) at this scale, same "fine for now" call already made for like/comment counts elsewhere in this codebase. Noted here as tech debt (see below) rather than silently skipped, per the phase spec.
- **RLS pattern (the rule)**: read = any authenticated user (`for select to authenticated using (true)`); write (insert/update/delete) = scoped to the owning row via `auth.uid()` matching that table's owner column. Applied to `profiles`, `instruments`, `genres`, `profile_instruments`, `profile_genres`, `media_posts`, `likes`, `comments`.
- **Deviations from that pattern** (private-by-nature data, not public-read):
  - `connections` — read scoped to `requester_id = auth.uid() or recipient_id = auth.uid()`; insert by requester only; update (accept only — see below) by recipient only; **delete by either party** (`connections_delete_own`, added in `0005_connections_delete_and_pair_constraint.sql`) — covers both the requester cancelling a pending request and either party removing an existing connection, one policy for both since it doesn't branch on `status`. Uniqueness is enforced direction-agnostically via `connections_unique_pair` (a `unique index` on `(least(requester_id, recipient_id), greatest(requester_id, recipient_id))`), not the original `unique(requester_id, recipient_id)` (dropped in the same migration) — that only blocked exact-direction duplicates, so B could previously insert a mirrored `B→A` row while `A→B` was still pending. `sendRequest()` in `src/lib/connections.ts` handles the resulting unique-violation (`23505`) by throwing "there's already a connection between you two" rather than assuming a fresh insert always succeeds. A `no_self_connect` CHECK (`requester_id <> recipient_id`) blocks self-connections — see the drift note below.
    - **Declining a pending request and removing an accepted connection are both a DELETE, never a status update.** `status = 'declined'` is a dead enum value going forward — no code path in the app writes it (`cancelOrDeclineOrRemove()` in `src/lib/connections.ts` is the single function behind cancel/decline/remove, all three being "delete this row"). The column still allows the DB value (dropping/redefining the enum wasn't worth the churn for a value nothing writes), and `getConnectionStatus()` defensively treats a `declined` row the same as no row at all if one is ever encountered, but new code should never set it.
    - **Drift found and fixed in Phase 3**: `no_self_connect` was already applied directly to both live databases in a prior session but was never captured in a migration file — found by querying `pg_constraint` on both projects during Phase 3's Step 1 confirmation, not assumed from the migration files. `0008_availability_status_and_self_connect_check.sql` formalizes it retroactively with a guarded `do $$ ... if not exists ... $$` block (safe on both current projects, which already have it, and on a hypothetical fresh project, which wouldn't).
  - `messages` — read scoped to `sender_id = auth.uid() or recipient_id = auth.uid()`.
    - **Insert is connection-gated (Phase 5a, `0011_messages_connection_gate_and_read_receipts.sql`)**: `messages_insert_own` now requires, in addition to `sender_id = auth.uid()`, an `accepted` row in `connections` between sender and recipient (checked direction-agnostically, same `least`/`greatest`-independent pattern as `connections_unique_pair`). A client-side "not connected" check would just be UI decoration; this is enforced at the DB level, per the phase spec. `src/lib/messages.ts`'s `sendMessage()` distinguishes this from a generic error by SQLSTATE `42501` (the standard Postgres code for an RLS `WITH CHECK` failure on INSERT) and throws a dedicated `NotConnectedError`.
    - **Update is scoped to marking read (Phase 5a)**: `messages_update_read` (`USING`/`WITH CHECK` both `recipient_id = auth.uid()`) lets only the recipient update a message, plus a `BEFORE UPDATE` trigger (`guard_message_read_update` → `messages_guard_read_update`) that blocks the UPDATE surface from being used for anything except a `read_at` `NULL` → non-`NULL` transition: any change to `content`/`sender_id`/`recipient_id`/`created_at`, or re-setting `read_at` once already non-null, raises and rolls back the statement. The trigger function is `SECURITY DEFINER` (needed since it's called mid-statement regardless of the invoking role) but has `EXECUTE` revoked from `PUBLIC`/`anon`/`authenticated` — trigger invocation itself doesn't need an EXECUTE grant (Postgres calls trigger functions directly), so this only blocks it from being called directly as an RPC (`/rest/v1/rpc/guard_message_read_update`), the same class of issue `dev_confirm_user_email` already had. Caught by the Supabase advisor after applying the migration, not anticipated up front — the phase spec's literal SQL didn't include this restriction.
    - **DELETE remains deliberately deferred, not an oversight.** Unsend/soft-delete is explicitly out of scope for Phase 5a (tracked below in Known tech debt) — messages accumulate with no way to remove a row from either the app or the RLS suite's fixtures.
    - **Realtime (Postgres Changes) is enabled on `messages`** (`alter publication supabase_realtime add table public.messages`) — the first real Realtime usage in this codebase (see Shared services and Testing). `supabase_realtime` was an empty publication before this.
- **`push_tokens`** (Phase 5b, `0012_push_notifications.sql`) — one row per `(profile_id, expo_push_token)` pair, `unique(profile_id, expo_push_token)` so a user's multiple devices don't collide, `platform` CHECK'd to `'ios'|'android'`. RLS is a single `push_tokens_owner_all` `FOR ALL` policy (`profile_id = auth.uid()`, both `USING`/`WITH CHECK`) — no client role, not even the owner reading *other* rows, has any broader access; the Edge Function reads across all profiles via its own service-role client, bypassing RLS entirely, never via a client-facing policy. `updated_at` maintained by the same shared `update_updated_at()` trigger as `profiles`/`connections`.
  - **First use of `pg_net` and Edge Functions in this codebase** (matches Phase 5a's own "first real Realtime usage" callout). Pattern: `AFTER INSERT` trigger → async `net.http_post` → Edge Function (`send-push-notification`) → Expo Push API — chosen over client-side-on-insert or a purely Realtime-driven approach because neither reliably fires for a recipient whose app isn't open, which is the entire point of push.
  - **`pg_net` is enabled but shows up on the security advisor as `extension_in_public`** ("Extension in Public... Move it to another schema") on both projects — confirmed this is **not actually fixable**: `alter extension pg_net set schema net;` errors with `cannot move extension "pg_net" into schema "net" because the extension contains the schema` (pg_net owns/creates the `net` schema itself as part of its own install; Postgres won't let you move an extension into a schema that's one of its own member objects). `net.http_post` etc. genuinely live in the `net` schema regardless (confirmed via `pg_proc`/`pg_namespace`, not assumed) — the advisor's generic "move it" remediation just doesn't apply cleanly to this specific extension. Disclosed as an accepted, unfixable advisory, same treatment as the other pre-existing ones below.
  - **One shared `notify_push_webhook()` trigger function for all four sources** (`messages`/`connections`/`likes`/`comments`) rather than four near-copies of the same "build a payload, call `net.http_post`" logic — branches on `TG_TABLE_NAME`. Fires on every `messages`/`likes`/`comments` insert; on `connections`, the trigger itself is scoped with `for each row when (NEW.status = 'pending')` (cheaper than an in-body check — skips invoking the function at all for a non-pending insert, confirmed via `net._http_response` row counts before/after that no call fires for an `accepted` insert). `SECURITY DEFINER` (needs to read `vault.decrypted_secrets` regardless of the inserting role) with `EXECUTE` revoked from `PUBLIC`/`anon`/`authenticated` — applied proactively this time, not caught after-the-fact by the advisor (see the new Phase-completion checklist line).
  - **Actor-exclusion is verified necessary, not just defensive.** Checked directly against live `pg_policies` (not assumed, per the phase spec's own instruction): `likes_insert_own`/`comments_insert_own` only check `user_id = auth.uid()` — nothing stops a user from liking or commenting on their *own* post today. So `notify_push_webhook()`'s `if v_actor_id = v_recipient_id then return NEW` check is load-bearing, not belt-and-suspenders; without it, self-likes/self-comments would self-notify.
  - **Webhook authentication is a Vault-backed shared secret, not `verify_jwt`.** `verify_jwt` only proves "the caller has some valid project key" — a signed-in user's own session JWT would pass it too, which doesn't prove "this came from our own trigger". Instead: a random secret (`push_webhook_secret`, generated server-side via `gen_random_bytes`, never written to any migration file or seen by whoever applies it) is created once via `vault.create_secret()` and sent by the trigger as a custom `x-webhook-secret` header; the function independently re-reads the same secret from Vault (via `get_push_webhook_secret()`, a `public`-schema wrapper since PostgREST doesn't expose the `vault` schema, `EXECUTE` restricted to `service_role` only) and compares. The Edge Function's own admin client needs no manually-configured secret at all — `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are auto-provisioned to every Edge Function by the platform. `verify_jwt: false` on the deployed function — matches Supabase's own documented pattern for service-to-service/external-webhook calls, not a loosened-for-convenience setting.
  - **Confirmed end-to-end against the live test project, not just deployed-and-assumed-working**: inserted real pending-connection/message/like/comment rows via direct SQL and read back `net._http_response` (status 200) after each; confirmed a real (fake but well-formed) Expo push token got a genuine `DeviceNotRegistered` ticket from Expo's actual API and was deleted from `push_tokens` by the function's own cleanup logic (not simulated — this was Expo's real API responding to a real HTTP call); confirmed a forged request with the wrong `x-webhook-secret` gets a real `401` via `curl` directly against the deployed function URL; confirmed an `accepted`-status connection insert produces zero new `net._http_response` rows (the `WHEN` clause actually skips it, not just in theory).
- **`profiles.availability_statuses`** (`availability_status[]`, `not null default '{}'`, added in `0008_availability_status_and_self_connect_check.sql`) — multi-select availability tags (`looking_for_band`, `available_for_session_work`, `open_to_auditions`, `forming_band`, `open_to_collabs`, `not_currently_looking`; see `AVAILABILITY_STATUSES` in `src/lib/types.ts` for labels). No RLS change needed — confirmed against the live `profiles_read`/`profiles_update_own` policy definitions on both projects (not assumed) that they already cover the whole row, this column included.
- **Naming**: snake_case columns throughout. The FK to the acting user is `profile_id` on tables tied to a profile-owned resource (`media_posts`, `profile_instruments`, `profile_genres`), but `user_id` on `likes`/`comments` (matches the Phase 2 spec's naming, still references `profiles(id)`) — this inconsistency is intentional-by-spec, not a bug; don't "fix" it without checking both.
- `updated_at` columns are maintained by the shared `update_updated_at()` trigger where present (`profiles`, `connections`); most tables don't need it and don't have the column.

## Storage
- Single bucket: **`media`** — holds both profile/onboarding intro media and feed post media.
- **`storage.buckets` needs its own SELECT policy, separate from the `storage.objects` policies.** The bucket and its objects policies (`media_public_read`/`media_user_upload`/etc.) were originally created by hand outside any migration; `storage.buckets` has RLS enabled with zero policies by default, so without one the Storage API can't resolve the bucket for an authenticated request and returns "Bucket not found" even though the bucket exists and the objects policies are correct — this broke posting/uploading media on production. Fixed by `0006_storage_buckets_select_policy.sql` (adds `buckets_read`), then `0007_media_storage_setup.sql` formalizes the whole bucket + all its object policies as one idempotent migration (`on conflict do nothing` / `drop policy if exists` + recreate) so a fresh project can be provisioned identically instead of drifting via manual dashboard setup, which is how this gap happened in the first place.
- Path convention:
  - Profile/onboarding intro media: `{user_id}/intro.{ext}` — always the same key (`upsert: true`), since a profile has exactly one intro slot.
  - Feed post media: `{user_id}/posts/{random_id}.{ext}`, thumbnail (video only) at `{user_id}/posts/{random_id}_thumb.jpg` — unique per post, since a user can have many posts.
- Per-media-type handling (all in `src/lib/mediaUpload.ts`):
  - **Photo**: resized to max 1600px on the long edge (only ever downsizes, never upscales small images) + recompressed to JPEG at 0.8 quality via `expo-image-manipulator`'s context-based API (`ImageManipulator.manipulate(uri).resize(...).renderAsync()` → `.saveAsync()` — the old `manipulateAsync` free function is deprecated as of the installed version).
  - **Video**: capped at 60s + medium quality via picker options only — no client-side transcoding.
  - **Audio**: capped at 60s via an auto-stop timer inside the recording hook.
  - **Video thumbnail**: generated via `expo-video-thumbnails` at `time: 0`, best-effort — upload proceeds without one if generation fails.
  - **Upload transport: ArrayBuffer, not Blob.** `uploadToStorage()` reads the local file via `expo-file-system`'s class-based `File` API (`new File(uri).arrayBuffer()`), not `fetch(uri).blob()`. `@supabase/storage-js`'s own docs state Blob/File/FormData "does not work as intended" in React Native — RN's Blob→FormData serialization doesn't reliably carry real bytes/content-type through, which surfaced as a real bug (`mime type text/plain is not supported` on every upload, since the mismatched Blob always fell back to `text/plain`). Content type is now passed explicitly per call site via `mimeTypeFor(ext)` rather than trusted from `blob.type`. **Confirmed fixed** — user re-tested a real post (photo + caption) in the simulator against production after this fix and it uploaded successfully; the earlier report of the same error on a first fix attempt (which only addressed the symptom, not the Blob/ArrayBuffer root cause) did not recur. Don't revert to `fetch().blob()` for storage uploads.

## Shared services
- `src/lib/supabase.ts` — the Supabase client singleton, typed via the generated `Database` type.
- `src/lib/database.types.ts` — **generated**, do not hand-edit. Regenerate after any schema change via the Supabase MCP `generate_typescript_types` tool (no local Supabase CLI — see Known tech debt).
- `src/lib/types.ts` — hand-written domain types: plain entity interfaces (`Profile`, `MediaPost`, `Like`, `Comment`, ...), and composed row types matching the exact shape of specific joined queries (`FullProfile`, `FeedPostRow`, `PostDetailRow`) — used against the relevant Supabase query rather than relying on postgrest-js's automatic embed inference, which doesn't handle multi-level nested embeds well.
  - **Type override pattern**: `PublicProfileScreen.tsx`/`MyProfileScreen.tsx` use `.returns<T>().single()` (returns called *before* single). As of the currently-installed `postgrest-js`, `.returns()` is deprecated in favor of `.overrideTypes<T, { merge: false }>()`, and the officially documented ordering is *after* `.single()`/`.maybeSingle()`, not before — the old ordering still happens to compile today but is fragile (a new test file using the identical-looking pattern hit a `never` type error that only resolved by switching to `.single().overrideTypes<T, { merge: false }>()`). Worth migrating the existing screens the same way in a future pass; use the new pattern for anything new.
- `src/lib/mediaUpload.ts` — owns **all** media picking, recording, compression, thumbnailing, and upload logic. Used by onboarding (`Step4Media`), My Profile's edit form, and Create Post.
- `src/lib/connections.ts` — the **only** place connection-status logic lives (Phase 3): `getConnectionStatus`, `sendRequest`, `acceptRequest`, `cancelOrDeclineOrRemove`, `listIncomingRequests`, `listSentRequests`, `listAcceptedConnections`. `PublicProfileScreen` (the connect button) and `ConnectionsScreen` (the three list sections) both consume it rather than duplicating queries. `getConnectionStatus` resolves direction-agnostically (one query checking both requester/recipient orderings, since `connections_unique_pair` guarantees at most one row per pair) and returns `{ status, connectionId }` — the id is included (a small extension beyond the phase spec's plain-string return) because the UI needs it to act on the row via the other functions; flagged here rather than silently deviating. `listIncomingRequests`/`listSentRequests` are both thin wrappers around one private `listPendingRequests(userId, direction)` — the two queries were identical except which column is filtered on and which joined profile is "the other person".
- `SKILL_LEVELS` (the beginner/intermediate/advanced/professional label list) lives in `src/lib/types.ts` next to `ExperienceLevel` — shared by `Step2Instruments.tsx` and `MyProfileScreen.tsx`'s edit form. `AVAILABILITY_STATUSES` (Phase 3) is the same pattern for `availability_statuses`, used by `MyProfileScreen`'s edit form and `ProfileBody`'s badge labels. Don't recreate either locally in a screen.
- `src/lib/discover.ts` (Phase 4b) — the **only** place the `discover_profiles` RPC is called from; `DiscoverScreen` consumes `discoverProfiles(filters)` rather than calling `supabase.rpc()` directly, same single-source-of-truth-per-concern pattern as `connections.ts`.
- `src/lib/messages.ts` (Phase 5a) — the **only** place messaging logic lives: `listConversations()`, `getThread(otherUserId, { before?, limit })`, `sendMessage(recipientId, content)`, `markThreadRead(otherUserId)`, `subscribeToThread(otherUserId, onInsert)`, plus `getUnreadCount()` (added beyond the phase spec's listed functions, for the Messages tab badge — same small, disclosed extension-beyond-spec pattern as `connections.ts`'s `connectionId` inclusion) and the `NotConnectedError` class `sendMessage` throws on an RLS rejection.
  - **Conversations are derived client-side, not stored.** `messages` has no `conversation_id` (see Data model) — `listConversations()` fetches the caller's full message history (sender-or-recipient, newest-first, both parties' profiles joined) and groups it by "the other participant" in a `Map` (insertion order = recency, since rows arrive newest-first and each participant's first-seen row is their most recent). Unbounded/no pagination — fine at this app's current scale, same "fine for now" call already made for like/comment counts; flagged in Known tech debt since a user with a large message history would need this capped.
  - **Every function resolves the current user internally** via `supabase.auth.getSession()` (a local, non-network read, not `getUser()`) rather than taking a `viewerId`/`userId` param — a deliberate deviation from the phase spec's own listed signatures (which omit one), flagged here per this repo's own convention for such judgment calls rather than silently diverging.
  - **`subscribeToThread`** registers two `postgres_changes` listeners on one channel (`sender_id=eq.<other>` and `recipient_id=eq.<other>`) instead of one filter, since Postgres Changes' `filter` only supports a single `column=operator.value` predicate, not an OR across two columns — each listener is still independently narrowed by `messages_read_own` RLS, so between the two, only messages actually involving the current user in either direction can ever arrive (confirmed via an integration test — see Testing — not just assumed).
  - **First real Realtime usage in this codebase** — `supabase_realtime` was an empty publication before Phase 5a (see Data model). Uses **Postgres Changes**, not Broadcast: RLS already filters correctly per-user via the existing `messages_read_own` policy, and this is a 1:1 thread view, not presence/typing indicators, so Postgres Changes' built-in RLS enforcement is the simpler fit — no reason to build a second, parallel authorization mechanism via Broadcast.
- `src/lib/pushNotifications.ts` (Phase 5b) — the **only** place push-token registration lives: `registerPushToken(profileId)` (requests permission if undecided, creates the Android notification channel, fetches an Expo push token, upserts it) and `subscribeToPushTokenChanges(profileId)` (the rare-token-refresh listener). `AppContext` calls both on the `authenticated` transition (see Navigation & state) rather than any screen calling them directly.
- `supabase/functions/send-push-notification/` (Phase 5b) — the **first Edge Function in this codebase**. Split into `index.ts` (the `Deno.serve` handler — env vars, the admin Supabase client, the webhook-secret check, calling Expo's push API) and `lib.ts` (pure payload-building/response-parsing logic, zero Deno-specific dependencies on purpose). That split exists specifically so `lib.ts` can be imported and unit-tested from this repo's existing Jest setup (`__tests__/unit/sendPushNotification.test.ts`) the same way every other shared-service module is tested — `index.ts` itself is never imported by Jest (Deno globals/`jsr:`/`npm:` specifiers aren't something Jest/babel resolves) and is excluded from the root `tsconfig.json`'s typecheck (`supabase/functions/**`) for the same reason. See Testing for what this does and doesn't let the automated suite verify.
- `src/components/` holds cross-screen reusable UI: `AudioPlayer`, `ProfileBody` (the full profile view, used by both `PublicProfileScreen` and `MyProfileScreen`; takes an optional `actionSlot` prop rendered in the header, below the experience badge — `PublicProfileScreen` uses it for the connect button, `MyProfileScreen` doesn't pass one), `VideoPlayerBlock` (used by `ProfileBody` and `PostDetailScreen`), `Avatar` (Phase 3 — the image-or-initial-circle pattern, `size: 'sm'|'md'|'lg'|'xl'`, used by `ProfileBody`, `ConnectionsScreen`, `FeedScreen`, and both author/comment-author spots in `PostDetailScreen`; it existed as five separate copy-pasted inline blocks before being extracted), `CityPicker` (Phase 4a — a controlled `{ city, state, cityId }` component backed by the `cities` table: search-and-select from the reference list, or a "my city isn't listed" free-text fallback that clears `cityId`; used by both `Step1BasicInfo` (onboarding) and `MyProfileScreen`'s edit form, the two places location is settable — built fresh since no prior onboarding step had a search-list pattern to follow), `ChipToggleGroup` + `toggleInSet` (promoted out of `MyProfileScreen.tsx` in Phase 4b, once `DiscoverScreen`'s instrument/genre filters needed the identical pill-toggle pattern that Phase 3 had already merged for Genres/Availability within that one file — exactly the "no duplicate implementations" case this rule exists for; `MyProfileScreen.tsx` now imports from the shared location instead of keeping its own copy). A screen file exporting a component another screen imports is exactly the pattern to avoid — if two screens need the same UI, it belongs here.
- **Rule**: no duplicate implementations of the same concern. Extend the shared module instead of writing a parallel one — this is exactly what Phase 2 fixed (onboarding and My Profile had copy-pasted upload logic before being consolidated).

## Navigation & state
- `AppContext` (`src/navigation/AppContext.tsx`) owns a single state machine, `appState`:
  - `loading` — initial, while `supabase.auth.getSession()` resolves (8s timeout falls back to `unauthenticated`).
  - `unauthenticated` — no session.
  - `onboarding` — session exists, but no row in `profiles` yet for that user.
  - `authenticated` — session exists and a `profiles` row exists.
  - Transitions are driven by `supabase.auth.onAuthStateChange`, plus a manual `refreshProfile()` call (used by `Step4Media` after onboarding completes) to re-check and flip to `authenticated` without waiting for an auth event.
  - **Push-token registration (Phase 5b) hooks the `authenticated` transition itself**, not app launch and not one specific onboarding screen — a dedicated `useEffect` calls `registerPushToken`/`subscribeToPushTokenChanges` (see Shared services) whenever `appState === 'authenticated'`. Deliberately not bolted onto `Step4Media`: hooking the state-machine transition instead of one onboarding screen is what makes "re-register on auth state change" (an existing user signing in on a *new* device, not just finishing onboarding for the first time) work for free, per the phase spec's own requirement.
  - **Checked, not assumed: no notification-tap-vs-Realtime double-handling is possible today.** `ThreadScreen`'s Realtime subscription (`src/lib/messages.ts`) only exists while that screen is focused; a notification tap brings the app forward and navigates in, at which point `ThreadScreen` does its own fresh `getThread()` fetch on mount that already includes whatever message triggered the notification — the Realtime subscription only starts contributing *after* that initial fetch, so there's no path where the same message could be both delivered via a notification-driven UI update and duplicated by the subscription. No de-dup logic was added because there is currently nothing to de-dup.
- `RootNavigator` picks one of three navigators based on `appState`:
  - **AuthStack**: `Login`, `SignUp`.
  - **OnboardingStack** (wrapped in `OnboardingProvider`, which holds the in-progress draft): `Step1`–`Step4`.
  - **MainStack** (top-level, `headerShown: false` except where overridden): `Tabs` (bottom tabs: `Feed`, `Discover`, `Connections`, `MyProfile`, `Messages`), `PublicProfile`, `PostDetail`, `CreatePost` (`presentation: 'modal'`).
  - **`Connections` tab** (Phase 3, `ConnectionsScreen.tsx`) — added as a 4th bottom tab rather than a header icon off Feed/MyProfile. Flagging this choice since the phase spec left it to judgment: Feed's tab has no header at all currently (`headerShown: false` at the `Tab.Navigator` level, see above), so a header-icon entry point would have meant adding a header to just one tab; a dedicated tab matches the existing all-tabs-equal-weight structure and needs no new pattern. Three sections inside it (Requests/Sent/Connections) via a pill-toggle segmented control, the same visual pattern as the skill-level toggle in `Step2Instruments`/`MyProfileScreen`'s edit form — not a new component, reused styling.
  - **`Discover` tab** (Phase 4b, `DiscoverScreen.tsx`) — added as a 2nd bottom tab, immediately after Feed. Same judgment-call reasoning as Connections' placement (the phase spec left ordering to judgment beyond "propose Feed, Discover, Connections, MyProfile, Messages"): Discover is a primary content-finding surface on par with Feed, not a settings/utility screen, so it gets equal tab weight rather than being buried behind a header icon — and placing it right after Feed keeps the two "browse content" tabs adjacent, ahead of the "manage relationships"/"manage self" tabs (Connections, MyProfile). Filters (radius preset row + instrument/genre `ChipToggleGroup`s) live in the results `FlatList`'s `ListHeaderComponent` rather than a separate collapsible panel — scrolls together with results, no new interaction pattern needed, consistent with keeping the radius control to fixed presets (not a continuous slider) per the phase spec.
  - **`Messages` tab** (Phase 5a) — the tab slot already existed in `MainTabParamList`/the tab bar (per earlier CONVENTIONS.md notes), but `MessagesScreen.tsx` was a placeholder stub with no data — confirmed by reading the file rather than assumed, per the phase spec's own instruction. Replaced with `ConversationsListScreen.tsx`. `MainStack` gained a new `Thread` screen (`{ otherUserId, otherProfile? }` — `otherProfile` is an optional pre-fetched `ProfileSummary` so navigating from a screen that already has it, which is every current call site, skips a redundant fetch; `ThreadScreen` falls back to fetching it itself if absent, e.g. a future deep link). `MainTabs` now also computes an unread-message badge (`tabBarBadge`), refreshed via a `postgres_changes` subscription scoped to `recipient_id = auth.uid()` (INSERT and UPDATE — the latter so marking a thread read elsewhere decrements it) plus a `tabPress` listener (since `MainTabs` itself never unmounts between tab switches, a plain `useFocusEffect` on the tab screen wouldn't fire on re-tapping an already-active tab).
  - **Entry point to start a new conversation**: the phase spec left it to judgment whether to hang this off a profile or the Connections list — `ConnectionsScreen`'s "Connections" section (accepted connections only, the only state a DM is actually allowed for) gained a "Message" button next to "Remove", navigating straight to `Thread`. `PublicProfileScreen`'s accepted-connection state was left as-is (no second entry point added) — one clear path was enough, and the phase spec didn't ask for both.
- Screens that navigate from a tab (e.g. `Feed`) to a `MainStack`-level screen (`PostDetail`, `CreatePost`) use the `CompositeScreenProps<BottomTabScreenProps<...>, NativeStackScreenProps<...>>` pattern — established in Phase 2 since no cross-navigator navigation existed before it.

Schema is kept in sync manually by applying every file in `supabase/migrations/` to it whenever a new one is added — there's no automatic sync, this is a real second database that can drift if a migration is applied to production and forgotten here.
  - Both projects have `dev_confirm_user_email` — see Data model; it wasn't in a tracked migration until `0003_dev_confirm_user_email.sql` was added specifically to make this project reproducible on both.
- **RLS/security suite** (`__tests__/integration/rls.test.ts`), against the **test** project, with disposable-but-fixed fixture accounts. See the file's own header comment for the full reasoning; the short version:
  - **Fixed fixture accounts, not per-run signups.** Supabase's auth email rate limit fires on every `signUp()` call — even calls immediately bypassed via the `dev_confirm_user_email` RPC still trigger it — and has been observed as tight as ~1 signup per ~60–90 minutes, **per project**, on the free tier (confirmed independently tight on both the production and dedicated test projects — it's a platform default, not something specific to one project). A suite that signs up fresh users every run is not CI-safe.
  - **Fixtures are bootstrapped via direct SQL, not signUp().** The three fixtures (`bandmate.rls.fixture.{a,b,c}@gmail.com`, password `BandmateTest!23456`) are inserted straight into `auth.users`/`auth.identities`/`public.profiles` via the MCP `execute_sql` tool (elevated access, separate from any anon-key client), completely bypassing GoTrue's rate-limited signup endpoint. The test file's `signInOrBootstrapFixture()` helper still has a `signUp()` fallback for a from-scratch project where fixtures don't exist yet, but the real, current fixtures on the test project were created this way — that fallback path is untested against the live rate limit and would need the same SQL-bootstrap treatment if it ever needs to run again (e.g. after resetting the test project).
  - **`auth.users.email_change` must be explicitly set to `''`, not left `NULL`, when inserting a user row directly via SQL.** It has no DB-level default despite being nullable, and GoTrue's Go driver can't scan a NULL there (`sql: Scan error on column index 8, name "email_change": converting NULL to string is unsupported`) — signs in fail with an opaque `500 Database error querying schema` until this is set. Found via `get_logs(service: "auth")`, not from the client error message.
  - **Fixture emails must be on a domain with valid MX records.** GoTrue rejects signups on domains it can't resolve mail for — fake/reserved TLDs (`.dev`, `example.com`) get `email address invalid`. `gmail.com` works fine without needing a real mailbox behind it (no email ever actually needs to arrive, since confirmation is bypassed).
  - **`connections` now has a DELETE policy** (`connections_delete_own`, see Data model). The suite still reuses one persistent `B<->C` row across runs for the original negative-path tests (looked up first, only inserted if missing), but Phase 3 added a `withDisposableConnection()` helper (inserts, runs the assertion, deletes in a `finally` regardless of outcome) for all the new positive-path cases — either-party delete on pending/accepted, recipient-only accept — so they don't touch the shared row or leave dangling rows that would trip `connections_unique_pair` on the next run. Also added: requester-only insert (third party can't impersonate), the `no_self_connect` CHECK, direction-agnostic duplicate rejection (`23505` both ways), and `availability_statuses` read-any/write-own coverage.
  - **Phase 4a added `cities` coverage**: authenticated read (public-read pattern), and insert/update/delete rejected for both `authenticated` and `anon`. No disposable-fixture helper needed — every write attempt is expected to fail and leave no row, so there's nothing to clean up.
  - **`messages` also has no DELETE policy** and accumulates one row per run — harmless (no uniqueness constraint), but real; a periodic manual cleanup is reasonable if the table gets noisy.
  - **Phase 5a's connection-gated INSERT policy broke this suite's original fixture message setup** (it inserted B->C, but B<->C is deliberately kept `pending` — see above — so it silently stopped being insertable). Fixed by creating a throwaway `accepted` A<->C connection just long enough to insert the fixture message, then deleting the connection (nothing about the message depends on the connection continuing to exist — the check only runs at insert time); the message itself is now A->C, so the "uninvolved third party" role in those two tests moved from A to B.
  - **Only 3 fixture accounts exist (A/B/C), and `messages.test.ts` (below) also needs disposable A<->B/A<->C connections for its own cases.** Jest runs integration test files in parallel workers by default, so both files briefly wanting the same pair (`connections_unique_pair`) is expected contention, not a bug — `withDisposableConnection()` here and the equivalent helpers in `messages.test.ts` both retry-with-backoff on a `23505` rather than failing outright.
- **`messages` suite** (`__tests__/integration/messages.test.ts`, Phase 5a), also against the **test** project, in its own file rather than folded into `rls.test.ts` (same "new feature domain → new file" precedent as `discover.test.ts`). Reuses the same three `bandmate.rls.fixture.*` accounts via a plain sign-in (fails fast if missing, same choice `discover.test.ts` already made for its own fixtures) and manages all its own connection state via disposable helpers, so it doesn't depend on `rls.test.ts`'s fixtures or run order. Covers: insert allowed/rejected across accepted/pending/declined/no-relationship pairs (asserting the `42501` SQLSTATE specifically, not just "some error"); recipient-can/sender-cannot/non-participant-cannot mark read; the trigger rejecting a `content`/`sender_id`/`recipient_id`/`created_at` change and a second `read_at` change once already set; and a Realtime case that **actually verifies** RLS applies to `postgres_changes` rather than assuming it (a client subscribes with no column filter at all — event/schema/table only — so RLS is the only thing that could be gating what it receives; a negative control confirms a message it's not party to never arrives, a positive control confirms the same subscription is genuinely live, not silently broken). - **`push_tokens` suite** (Phase 5b, folded into `rls.test.ts` rather than a new file — unlike `messages`, this is a plain owner-only CRUD table with no complex enough surface to warrant its own file, same call already made for `cities`/`availability_statuses`). ## Known tech debt / deferred decisions
- **No EAS project exists for this app yet** (Phase 5b) — `app.json` has no `extra.eas.projectId`, which `expo-notifications`' `getExpoPushTokenAsync()` requires. `registerPushToken()` (see Shared services) checks for it and silently no-ops if it's missing, so this doesn't crash anything, but it means **push tokens cannot actually be obtained on any device, real or simulated, until `eas init` (or the Expo dashboard) creates one and populates that field** — not something creatable via any tool available this session (needs an interactive Expo account login). The whole server-side pipeline (trigger → Edge Function → Expo API) is confirmed working end-to-end regardless (see Data model) — this gap is purely "no device has ever registered a real token yet," not a bug in that pipeline.
- **No custom notification icon/sound assets** — `app.json`'s `expo-notifications` plugin config has neither (would need a 96×96 all-white transparent PNG for Android per the SDK 57 docs); notifications will render with whatever Android's default icon behavior falls back to until one's added. Low-priority visual polish, not a functional gap.
- **`EXPO_ACCESS_TOKEN` / Expo "enhanced security" status is unverified** — `send-push-notification/index.ts` reads `Deno.env.get('EXPO_ACCESS_TOKEN')` and includes it as a bearer token if present, but whether this Expo project actually needs one can't be checked without an EAS project existing first (see above). Revisit once one does.
- **`cities` table is still small** (4 rows as of Phase 4b — see Data model): enough to correctly exercise the picker/backfill/RLS/radius mechanics, not enough for rich real-world discovery variety. No spatial index yet either (same "fine at current scale" call as below); both worth revisiting once the table actually grows.
- **Discovery v1 scope, deferred by the Phase 4b spec, not forgotten**: `profile_instruments.skill_level` isn't a filter dimension (exists in the schema, just not wired into `discover_profiles`'s filtering — only used for display); search only ever centers on the caller's own `matched_city_id` ("near me"), there's no way to search centered on a different city (e.g. scouting a campus before moving); endorsements remain out of scope, unchanged since Phase 3.
- **`postgis` remains available-but-not-installed** on both projects (confirmed via `list_extensions`) — deliberately unused; `discover_profiles` uses `earthdistance`/`cube` instead (see Data model) so there's only one way to do distance math in this codebase, not two redundant ones.
- **Feed video rendering**: Feed cards show a thumbnail + play icon only, never a live video player — intentional, to avoid instantiating many video players in a scrolling list. Full playback only happens on Post Detail. Don't "fix" this by making Feed autoplay video.
- **Like/comment counts** are computed client-side from full embedded arrays (`likes(user_id)`, `comments(id)`), not database aggregates. Fine at current scale; will need count aggregates or an RPC if a post's like count grows large.
- **Post Detail interactions not automation-verified**: the like-toggle and Send-comment buttons were not confirmed via UI automation (simulator has no synthetic-touch API, and blind coordinate tapping proved unreliable at that precision). The code path is verified by type-checking, live schema validation, and structural parity with Feed's already-verified like-toggle — but do a manual pass before shipping.
- **Phase 5a: Messages/Connections screen interactions past the first row-tap not automation-verified**, same root cause as Post Detail above plus a new wrinkle — macOS's accessibility ("assistive access") permission for the AppleScript/`osascript`-based tap simulation used this session became unreliable mid-session (worked repeatedly, then started failing with `-25211 not allowed assistive access` with no code change on either side), an environment/OS-level issue outside this repo. What **was** confirmed live against the seeded test project: sign-in as `seed.user.0`, Feed loading, bottom-tab navigation (Feed→Messages, Feed→Connections) working repeatedly, the Messages tab's unread badge showing the correct seeded count ("1"), and `ConversationsListScreen` rendering the real seeded thread (Delilah Herzog / `seed_user_3`, correct last-message preview, timestamp, unread badge) — strong evidence the data/service layer (`listConversations()`) and list screen are correct end-to-end against a live Realtime-enabled Supabase project. **Not confirmed live**: opening a thread (`ThreadScreen`), sending a message, the Connections tab's segmented-control pills, and the "Message" button — do a manual pass on these before shipping.
- **`dev_confirm_user_email` RPC**: bypasses email confirmation, gated behind `__DEV__` in `SignUpScreen`. Remove/disable before any real email-confirmation flow ships. **Mitigated but not fully closed** (`0004_restrict_dev_confirm_user_email.sql`): still callable by `anon` (unavoidable — every call site calls it pre-session, see Testing) and still `SECURITY DEFINER`, but now scoped to rows created in the last 5 minutes and no longer callable by `authenticated`/`PUBLIC`. Removing this function entirely, not just narrowing it, is still the real fix once email confirmation ships for real.
- **Pre-existing security advisories** (not introduced by Phase 2, not yet addressed): `update_updated_at()` has a mutable search_path; the `media` bucket allows public listing; leaked-password protection is disabled. (`dev_confirm_user_email`'s anon-callable/SECURITY DEFINER advisory is now mitigated, not resolved — see above.) None are blocking for development; worth a pass before production.
- **No local Supabase CLI**: schema changes are applied and types regenerated via the Supabase MCP tools (`apply_migration`, `generate_typescript_types`), not `supabase` CLI commands. If the CLI gets installed later, prefer it for local dev workflows per the MCP server's own guidance, but keep migration files as the source of truth either way.
- **No Supabase branching**: requires a Pro-plan upgrade (real recurring cost); surfaced to the user, deferred. A second free-tier project (see Testing) substitutes for it — real isolation from production, but manual schema sync instead of automatic branch-from-parent. Revisit if the plan is ever upgraded — real branch-per-run testing would be strictly better once available.
- **Deprecated `.returns<T>().single()` ordering** still in use in `PublicProfileScreen.tsx`/`MyProfileScreen.tsx` (see Shared services above) — compiles fine today, but is a fragile, deprecated pattern. Migrate to `.single().overrideTypes<T, { merge: false }>()` in a future pass.

Still periodic/manual, still not urgent.
- **Running `npm run test:integration` (or `jest --selectProjects integration` without a specific test-name filter) executes `signup-onboarding.test.ts` too**, which does a real `signUp()` and burns the test project's auth rate limit. To re-run only the rate-limit-safe suites (`rls.test.ts`, `discover.test.ts`), pin the test name as well as the path(s) (`npx jest --selectProjects integration -t "" __tests__/integration/rls.test.ts __tests__/integration/discover.test.ts`) — the path argument alone was observed to still pick up every file in this project's Jest config.
- **Migration *history* can also drift independently of actual schema state**: found during Phase 3's Step 1 confirmation — production's `apply_migration` history is missing `0003_dev_confirm_user_email.sql`'s entry (though the function exists live, so it was applied directly via SQL at some point rather than through the tracked migration path) and the test project's history is missing `0006_storage_buckets_select_policy.sql`'s entry (though `0007`, applied after it, idempotently recreates the same policy, so no functional drift). Both confirmed harmless by checking live `pg_proc`/`pg_policies` state, not just `list_migrations`. Not fixed (rewriting migration history isn't something `apply_migration` supports), just disclosed — a reminder that `list_migrations` alone isn't sufficient to confirm two projects match; check live schema state too when in doubt.
- **Test project schema can drift from production**: applying a new migration only updates whichever project you point `apply_migration` at — there's no automation making that happen on both. **Partially mitigated**: `scripts/check-migrations-parity.js` runs in CI before the integration suite and fails loudly if a table/column exists on one project but not the other (see Testing). It does *not* catch everything — functions are an explicit, disclosed coverage gap (can't be checked without a service-role key), and it only runs in CI, not automatically when you apply a migration locally. Still always apply new migrations to both projects; the check is a safety net for when that's forgotten, not a substitute for doing it.
