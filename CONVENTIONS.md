# Bandmate — Conventions & Architecture

_Last updated: 2026-07-17 (Testing Setup). Update this file whenever a phase introduces or changes anything below — it should never go stale._

## Stack
- Expo SDK ~57.0.6, React Native 0.86.0, React 19.2.3, TypeScript ~6.0.3, strict mode on.
- **NativeWind v4** — Tailwind utility classes for RN styling, kept for a single mental model across future web/native work.
- **React Navigation v7** (`@react-navigation/native-stack` + `@react-navigation/bottom-tabs`) — stack-inside-tabs composition; see Navigation & state below.
- **Supabase** (`@supabase/supabase-js`) — Postgres + Auth + Storage backend, single project (`ktfrsgffgzepadmoryps`, ref also in `.env`).
- **Managed workflow** — no `ios`/`android` folders checked in (gitignored, regenerated on demand). All native modules used so far (`expo-audio`, `expo-video`, `expo-image-picker`, `expo-image-manipulator`, `expo-video-thumbnails`, `expo-asset`) are bundled in Expo Go for SDK 57, so the app runs there directly.
  - **What would force a custom dev client**: adding any native module *not* included in Expo Go — a custom Expo Modules API module, a third-party RN library without Expo Go support, background tasks/notifications requiring native entitlements, or anything needing a custom `Info.plist`/`AndroidManifest.xml` entry beyond what `app.json` config plugins cover.

## Data model
- Migrations live in `supabase/migrations/`, named `NNNN_description.sql` (`0001_init.sql`, `0002_media_feed.sql`, `0003_dev_confirm_user_email.sql`, ...). Applied via the Supabase MCP `apply_migration` tool (no local Supabase CLI installed — see Known tech debt) to **both** the production and test projects (see Testing) — there are two live databases now, and this is the only source of truth kept in sync between them.
- **RLS pattern (the rule)**: read = any authenticated user (`for select to authenticated using (true)`); write (insert/update/delete) = scoped to the owning row via `auth.uid()` matching that table's owner column. Applied to `profiles`, `instruments`, `genres`, `profile_instruments`, `profile_genres`, `media_posts`, `likes`, `comments`.
- **Deviations from that pattern** (private-by-nature data, not public-read):
  - `connections` — read scoped to `requester_id = auth.uid() or recipient_id = auth.uid()`; insert by requester only; update (accept/decline) by recipient only.
  - `messages` — read/insert scoped to `sender_id = auth.uid() or recipient_id = auth.uid()`.
- **Naming**: snake_case columns throughout. The FK to the acting user is `profile_id` on tables tied to a profile-owned resource (`media_posts`, `profile_instruments`, `profile_genres`), but `user_id` on `likes`/`comments` (matches the Phase 2 spec's naming, still references `profiles(id)`) — this inconsistency is intentional-by-spec, not a bug; don't "fix" it without checking both.
- `updated_at` columns are maintained by the shared `update_updated_at()` trigger where present (`profiles`, `connections`); most tables don't need it and don't have the column.

## Storage
- Single bucket: **`media`** — holds both profile/onboarding intro media and feed post media.
- Path convention:
  - Profile/onboarding intro media: `{user_id}/intro.{ext}` — always the same key (`upsert: true`), since a profile has exactly one intro slot.
  - Feed post media: `{user_id}/posts/{random_id}.{ext}`, thumbnail (video only) at `{user_id}/posts/{random_id}_thumb.jpg` — unique per post, since a user can have many posts.
- Per-media-type handling (all in `src/lib/mediaUpload.ts`):
  - **Photo**: resized to max 1600px on the long edge (only ever downsizes, never upscales small images) + recompressed to JPEG at 0.8 quality via `expo-image-manipulator`'s context-based API (`ImageManipulator.manipulate(uri).resize(...).renderAsync()` → `.saveAsync()` — the old `manipulateAsync` free function is deprecated as of the installed version).
  - **Video**: capped at 60s + medium quality via picker options only — no client-side transcoding.
  - **Audio**: capped at 60s via an auto-stop timer inside the recording hook.
  - **Video thumbnail**: generated via `expo-video-thumbnails` at `time: 0`, best-effort — upload proceeds without one if generation fails.

## Shared services
- `src/lib/supabase.ts` — the Supabase client singleton, typed via the generated `Database` type.
- `src/lib/database.types.ts` — **generated**, do not hand-edit. Regenerate after any schema change via the Supabase MCP `generate_typescript_types` tool (no local Supabase CLI — see Known tech debt).
- `src/lib/types.ts` — hand-written domain types: plain entity interfaces (`Profile`, `MediaPost`, `Like`, `Comment`, ...), and composed row types matching the exact shape of specific joined queries (`FullProfile`, `FeedPostRow`, `PostDetailRow`) — used against the relevant Supabase query rather than relying on postgrest-js's automatic embed inference, which doesn't handle multi-level nested embeds well.
  - **Type override pattern**: `PublicProfileScreen.tsx`/`MyProfileScreen.tsx` use `.returns<T>().single()` (returns called *before* single). As of the currently-installed `postgrest-js`, `.returns()` is deprecated in favor of `.overrideTypes<T, { merge: false }>()`, and the officially documented ordering is *after* `.single()`/`.maybeSingle()`, not before — the old ordering still happens to compile today but is fragile (a new test file using the identical-looking pattern hit a `never` type error that only resolved by switching to `.single().overrideTypes<T, { merge: false }>()`). Worth migrating the existing screens the same way in a future pass; use the new pattern for anything new.
- `src/lib/mediaUpload.ts` — owns **all** media picking, recording, compression, thumbnailing, and upload logic. Used by onboarding (`Step4Media`), My Profile's edit form, and Create Post.
- **Rule**: no duplicate implementations of the same concern. Extend the shared module instead of writing a parallel one — this is exactly what Phase 2 fixed (onboarding and My Profile had copy-pasted upload logic before being consolidated).

## Navigation & state
- `AppContext` (`src/navigation/AppContext.tsx`) owns a single state machine, `appState`:
  - `loading` — initial, while `supabase.auth.getSession()` resolves (8s timeout falls back to `unauthenticated`).
  - `unauthenticated` — no session.
  - `onboarding` — session exists, but no row in `profiles` yet for that user.
  - `authenticated` — session exists and a `profiles` row exists.
  - Transitions are driven by `supabase.auth.onAuthStateChange`, plus a manual `refreshProfile()` call (used by `Step4Media` after onboarding completes) to re-check and flip to `authenticated` without waiting for an auth event.
- `RootNavigator` picks one of three navigators based on `appState`:
  - **AuthStack**: `Login`, `SignUp`.
  - **OnboardingStack** (wrapped in `OnboardingProvider`, which holds the in-progress draft): `Step1`–`Step4`.
  - **MainStack** (top-level, `headerShown: false` except where overridden): `Tabs` (bottom tabs: `Feed`, `MyProfile`, `Messages`), `PublicProfile`, `PostDetail`, `CreatePost` (`presentation: 'modal'`).
- Screens that navigate from a tab (e.g. `Feed`) to a `MainStack`-level screen (`PostDetail`, `CreatePost`) use the `CompositeScreenProps<BottomTabScreenProps<...>, NativeStackScreenProps<...>>` pattern — established in Phase 2 since no cross-navigator navigation existed before it.

## Testing
- **Jest via two projects** (`package.json`'s `jest.projects`), not one config, because they need genuinely different environments:
  - **`unit`** (`__tests__/unit/**`, run via `npm test`) — `jest-expo` preset, RN-mocked environment (including a mocked `fetch` that never hits the network). Covers component/hook logic and `src/lib/mediaUpload.ts` / `src/lib/supabase.ts` via `jest.mock()` on every native/Supabase dependency.
  - **`integration`** (`__tests__/integration/**`, run via `npm run test:integration`) — plain Node environment with a dedicated `babel.integration.config.js` (deliberately *not* `babel-preset-expo`, which rewrites `process.env.EXPO_PUBLIC_*` into an `expo/virtual/env` import that only resolves inside Metro and breaks under Jest). Hits a real Supabase project over the network — real `fetch`, no mocks. Needs `ws` supplied as the Realtime transport (`realtime: { transport: WebSocket as WebSocketLikeConstructor }` from `@supabase/realtime-js`) purely so `createClient()` doesn't throw on Node 20 (no native `WebSocket`) — Realtime itself is never used.
  - `npm run test:all` runs both. CI (`.github/workflows/test.yml`) runs `unit` then `integration` on every push/PR to `main`.
- **Two separate Supabase projects — never mix them up:**
  - **Production** (`ktfrsgffgzepadmoryps`) — what the app itself connects to, via `EXPO_PUBLIC_SUPABASE_URL`/`EXPO_PUBLIC_SUPABASE_ANON_KEY` in `.env`. `src/lib/supabase.ts` reads only these two vars — nothing about the app's own connection changed when the test project was added.
  - **Test** (`hgpsqjwghaujaxeumisl`, "BandMate test") — a dedicated free-tier project that exists solely for `__tests__/integration/**`, via `TEST_SUPABASE_URL`/`TEST_SUPABASE_ANON_KEY` (also in `.env`, also read by both integration test files — never by any app code). Created because Supabase branching needs a Pro-plan upgrade (real recurring cost, surfaced to and deferred by the user). Schema is kept in sync manually by applying every file in `supabase/migrations/` to it whenever a new one is added — there's no automatic sync, this is a real second database that can drift if a migration is applied to production and forgotten here.
  - Both projects have `dev_confirm_user_email` — see Data model; it wasn't in a tracked migration until `0003_dev_confirm_user_email.sql` was added specifically to make this project reproducible on both.
- **RLS/security suite** (`__tests__/integration/rls.test.ts`), against the **test** project, with disposable-but-fixed fixture accounts. See the file's own header comment for the full reasoning; the short version:
  - **Fixed fixture accounts, not per-run signups.** Supabase's auth email rate limit fires on every `signUp()` call — even calls immediately bypassed via the `dev_confirm_user_email` RPC still trigger it — and has been observed as tight as ~1 signup per ~60–90 minutes, **per project**, on the free tier (confirmed independently tight on both the production and dedicated test projects — it's a platform default, not something specific to one project). A suite that signs up fresh users every run is not CI-safe.
  - **Fixtures are bootstrapped via direct SQL, not signUp().** The three fixtures (`bandmate.rls.fixture.{a,b,c}@gmail.com`, password `BandmateTest!23456`) are inserted straight into `auth.users`/`auth.identities`/`public.profiles` via the MCP `execute_sql` tool (elevated access, separate from any anon-key client), completely bypassing GoTrue's rate-limited signup endpoint. The test file's `signInOrBootstrapFixture()` helper still has a `signUp()` fallback for a from-scratch project where fixtures don't exist yet, but the real, current fixtures on the test project were created this way — that fallback path is untested against the live rate limit and would need the same SQL-bootstrap treatment if it ever needs to run again (e.g. after resetting the test project).
  - **`auth.users.email_change` must be explicitly set to `''`, not left `NULL`, when inserting a user row directly via SQL.** It has no DB-level default despite being nullable, and GoTrue's Go driver can't scan a NULL there (`sql: Scan error on column index 8, name "email_change": converting NULL to string is unsupported`) — signs in fail with an opaque `500 Database error querying schema` until this is set. Found via `get_logs(service: "auth")`, not from the client error message.
  - **Fixture emails must be on a domain with valid MX records.** GoTrue rejects signups on domains it can't resolve mail for — fake/reserved TLDs (`.dev`, `example.com`) get `email address invalid`. `gmail.com` works fine without needing a real mailbox behind it (no email ever actually needs to arrive, since confirmation is bypassed).
  - **`connections` has no DELETE policy and a `unique(requester_id, recipient_id)` constraint** — can't be recreated fresh every run like the other test data. The suite looks up an existing row between the fixtures first and reuses it; only creates one if missing.
  - **`messages` also has no DELETE policy** and accumulates one row per run — harmless (no uniqueness constraint), but real; a periodic manual cleanup is reasonable if the table gets noisy.
  - `media_posts` (and its cascaded `likes`/`comments`) *does* have a delete policy and is created + deleted fresh every run.
- **Critical-path suite** (`__tests__/integration/signup-onboarding.test.ts`), also against the **test** project: tests the signup flow itself, so — unlike the RLS suite — it inherently cannot be redesigned around fixed fixtures or SQL bootstrap; it calls the real `signUp()` API, once per run, because that's the exact thing being verified. This means it **will** hit the same auth rate limit if run too frequently (including via CI on frequent pushes to `main`) — an accepted, disclosed limitation of testing a live signup flow, not a bug. It also can't clean up its own `auth.users` row afterward (no service-role key available client-side — see below), so test accounts (`bandmate.critpath.*@gmail.com`) accumulate on the test project and need periodic manual deletion.
- **No service-role key available anywhere** — the Supabase MCP server only exposes anon/publishable keys (`get_publishable_keys`), by design. This is why: (a) fixture bootstrap uses direct SQL via the MCP `execute_sql` tool's own elevated access rather than an admin API call, (b) test cleanup that needs elevated privileges (deleting `auth.users` rows) is done the same way, manually in an interactive session, not automatically in CI. If a service-role key is ever added as a GitHub Actions secret for the test project, the critical-path test could switch to `supabase.auth.admin.deleteUser()` in an `afterAll` for real per-run cleanup.
- **CI secrets required** — the `integration` GitHub Actions job needs `TEST_SUPABASE_URL` and `TEST_SUPABASE_ANON_KEY` set as repo secrets (Settings → Secrets and variables → Actions), pointing at the **test** project (`hgpsqjwghaujaxeumisl`), not production. Same values as the local `.env` (gitignored, values not repeated here even though the anon/publishable key is RLS-protected by design — same no-committed-keys-anywhere standard applied to the production project's credentials).
- **Repo default branch**: pushed as `main` (workflow triggers on `main`). A stray `master` branch may still exist on GitHub from the very first push (GitHub set it as default since it landed first, and it couldn't be deleted without changing the default branch first, which needs dashboard access this session didn't have) — safe to delete `master` and confirm `main` is default via GitHub's UI whenever convenient; both point at the same commit.

## Known tech debt / deferred decisions
- **Feed video rendering**: Feed cards show a thumbnail + play icon only, never a live video player — intentional, to avoid instantiating many video players in a scrolling list. Full playback only happens on Post Detail. Don't "fix" this by making Feed autoplay video.
- **Like/comment counts** are computed client-side from full embedded arrays (`likes(user_id)`, `comments(id)`), not database aggregates. Fine at current scale; will need count aggregates or an RPC if a post's like count grows large.
- **Post Detail interactions not automation-verified**: the like-toggle and Send-comment buttons were not confirmed via UI automation (simulator has no synthetic-touch API, and blind coordinate tapping proved unreliable at that precision). The code path is verified by type-checking, live schema validation, and structural parity with Feed's already-verified like-toggle — but do a manual pass before shipping.
- **`dev_confirm_user_email` RPC**: bypasses email confirmation, gated behind `__DEV__` in `SignUpScreen`. Remove/disable before any real email-confirmation flow ships.
- **Pre-existing security advisories** (not introduced by Phase 2, not yet addressed): `update_updated_at()` has a mutable search_path; the `media` bucket allows public listing; `dev_confirm_user_email` is callable by `anon`/`authenticated` as `SECURITY DEFINER`; leaked-password protection is disabled. None are blocking for development; worth a pass before production.
- **No local Supabase CLI**: schema changes are applied and types regenerated via the Supabase MCP tools (`apply_migration`, `generate_typescript_types`), not `supabase` CLI commands. If the CLI gets installed later, prefer it for local dev workflows per the MCP server's own guidance, but keep migration files as the source of truth either way.
- **No Supabase branching**: requires a Pro-plan upgrade (real recurring cost); surfaced to the user, deferred. A second free-tier project (see Testing) substitutes for it — real isolation from production, but manual schema sync instead of automatic branch-from-parent. Revisit if the plan is ever upgraded — real branch-per-run testing would be strictly better once available.
- **Deprecated `.returns<T>().single()` ordering** still in use in `PublicProfileScreen.tsx`/`MyProfileScreen.tsx` (see Shared services above) — compiles fine today, but is a fragile, deprecated pattern. Migrate to `.single().overrideTypes<T, { merge: false }>()` in a future pass.
- **Integration test accounts accumulate on the test project** (never production — confirmed clean, see Testing): `bandmate.rls.fixture.{a,b,c}@gmail.com` are permanent fixtures (intentional). `bandmate.critpath.*@gmail.com` accounts from the critical-path suite are not — each run adds one, undeleted (no service-role key to clean up with). Periodic manual cleanup via SQL (`delete from auth.users where email like 'bandmate.critpath.%'`) is reasonable maintenance, not urgent.
- **Test project schema can drift from production**: applying a new migration only updates whichever project you point `apply_migration` at. There's no automation keeping `hgpsqjwghaujaxeumisl` in sync with `ktfrsgffgzepadmoryps` — a migration applied to one and forgotten on the other will make the integration suite pass/fail against a schema the app doesn't actually run against. Always apply new migrations to both.

## Phase log
- Phase 0 — Initial Expo + TypeScript scaffold.
- Phase 1 — Auth (email/password), 4-step onboarding, profile view/edit; initial schema (`profiles`, `instruments`, `genres`, `media_posts`, `connections`, `messages`) with RLS.
- Phase 2 — Media feed: real generated Supabase types wired in, `likes`/`comments` tables added, upload logic consolidated into `src/lib/mediaUpload.ts`, chronological Feed with like/comment counts, Create Post flow, Post Detail with comments.
- Testing Setup — Jest (unit + live-network integration projects) and React Native Testing Library; RLS/security suite covering all 10 tables with fixed fixture accounts; unit tests for `mediaUpload.ts`/`supabase.ts`; critical-path signup→onboarding→profile test; GitHub Actions CI on push/PR to `main`; first push of the whole app to `github.com/olearyld/BandMate-2.0`; dedicated free-tier test Supabase project (`hgpsqjwghaujaxeumisl`) added and integration suite repointed at it, keeping production untouched by test data.
